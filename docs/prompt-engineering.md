# Prompt Engineering Strategy

## System Prompt Structure

Each persona's system prompt follows a consistent layered structure:

```
1. IDENTITY BLOCK    — who the person is, background, credentials
2. STYLE BLOCK       — language patterns, vocabulary, code-switching rules
3. TEACHING BLOCK    — pedagogical approach, methodology
4. EXAMPLES BLOCK    — sample phrases to calibrate tone (few-shot examples)
5. CONSTRAINTS BLOCK — what NOT to do (break character, be robotic, etc.)
```

This layered approach ensures the model has both the persona identity (what) and the behavioral rules (how) clearly separated.

---

## Design Decisions

### 1. Identity First
The system prompt opens with a rich identity paragraph. This grounds the model in a specific person before adding stylistic rules. Without this anchor, the model tends to drift into a generic "friendly teacher" persona.

### 2. Explicit Vocabulary Lists
For Hitesh's Hinglish style, providing explicit word lists (e.g., "hanji", "bilkul", "samjha?") with translations prevents the model from either over-doing or under-doing the code-switching. The translation in parentheses prevents incorrect usage.

### 3. "Use Naturally, Not Robotically"
Both prompts include the instruction to use phrases naturally, not mechanically. Without this guardrail, the model tends to start every sentence with the same phrase, which feels artificial.

### 4. Example Phrases as Few-Shot Calibration
Including 5–6 sample phrases acts as implicit few-shot examples of the correct tone and language blend. This is more effective than describing the tone abstractly.

### 5. WHY-HOW-OHH Pattern (Piyush)
Piyush's teaching pattern is explicitly encoded as a numbered sequence: start with WHY → explain HOW → lead to the OHH moment. This guides the model to structure responses in a way that matches Piyush's actual pedagogical style.

### 6. Temperature Settings
- **Hitesh: 0.85** — slightly higher temperature produces more expressive, varied Hinglish with cultural references, which matches his energetic teaching style
- **Piyush: 0.80** — slightly lower produces more precise, structured answers that align with his direct, opinion-driven style

### 7. Backend Prompt Injection
The system prompt is injected exclusively on the backend. The frontend never touches it. This serves two purposes:
- **Security**: The API key and prompt details are never exposed to the browser
- **Consistency**: The persona cannot be tampered with via developer tools or custom API calls

## Context Management

### Stateless Backend Design
The backend holds no conversation state. The frontend sends the full `messages[]` array with every request. This simplifies scaling and avoids session management complexity.

### Message Window Trimming
The backend trims incoming messages to the last 20 before sending to OpenAI:

```js
function trimMessages(messages, maxCount) {
  if (messages.length <= maxCount) return messages;
  return messages.slice(messages.length - maxCount);
}
```

**Why 20 messages?**  
- 20 messages ≈ 10 conversation turns
- Typical educational chat rarely needs more than 5–6 turns of active context
- gpt-4o-mini's 128k context window is large, but cost grows with context length
- 20 messages keeps costs low while preserving conversational coherence

### System Prompt Always First
The system prompt is prepended as `{ role: 'system', content: systemPrompt }` before the trimmed messages. This means the persona identity is never trimmed away, even in very long conversations.

### Greeting Message
The persona greeting is sent as a synthetic assistant message at conversation start. It is filtered out of the API payload (it was generated by us, not the LLM) to avoid confusing the model with a "ghost" message, but appears in the UI to give the conversation a warm, on-brand start.

## Handling Persona Switching

When the user switches personas:
1. The frontend confirms with the user (to avoid accidental conversation loss)
2. The `AbortController` aborts any in-flight SSE stream
3. The `messages` state is cleared and replaced with only the new persona's greeting
4. The `currentPersona` state updates, which propagates the new `personaId` to all future API calls

This ensures clean separation between persona sessions.

## SSE Streaming Design

### Why SSE over WebSockets?
Server-Sent Events are a simpler fit for this use case:
- Unidirectional (server → client) — exactly what we need for streaming responses
- Native browser support via `fetch` + `ReadableStream`
- HTTP/1.1 compatible, no upgrade handshake
- Works naturally through Vite's dev proxy

### Buffer Strategy
Raw SSE data arrives in chunks that may split across events. The reader accumulates a buffer and splits on `\n\n` (SSE event boundary):

```js
buffer += decoder.decode(value, { stream: true });
const parts = buffer.split('\n\n');
buffer = parts.pop(); // keep incomplete trailing chunk
```

The `{ stream: true }` flag in the decoder handles multi-byte characters that may split across chunks.

### Abort / Stop Generation
An `AbortController` is attached to each fetch. Clicking "Stop" calls `controller.abort()`, which triggers an `AbortError` in the reader loop. The `finally` block resets `isStreaming`, and the partial assistant message is preserved in the UI.
